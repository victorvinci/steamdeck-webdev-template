### Added

- **Added** `scripts/rename-template.sh` + `docs/FORK.md` — fork-onboarding pair for the v1.0.0 template-usability pass. The script mechanizes the four-category rename (project name, GitHub owner, npm scope, maintainer email) across an explicit allowlist of files; historical records (`CHANGELOG.md`, `.ai-attribution.jsonl`, `docs/SECURITY-AUDIT-v1.0.0*.md`) are left alone. Validates input shapes, requires a clean worktree unless `--force-dirty`, runs `npm run format` + `npm run check` before handing back. `docs/FORK.md` documents the full 10-step onboarding: rename, manual file cleanups the script can't automate, GitHub Settings (Pages, package write, private vulnerability reporting), branch rulesets, external services (Nx Cloud, Renovate), signed commits, first CI run, and a pruning checklist for boilerplate that new forks won't need. `README.md` Table of Contents and a top-of-README callout point at the doc so fresh forks land on it before running Quick Start.
- **Added** `apps/frontend-e2e/src/users.failure.spec.ts` — Playwright coverage for the three `/users` failure states that the seeded happy-path suite can't reach. Uses `page.route` to stub `**/api/users*` at the network layer (no backend or DB changes): 503 surfaces the `role="alert"` UI with Retry button, empty `{users:[],total:0}` renders `No users yet.`, and a counter-based handler fails the first call then seeds the second so the Retry button's refetch path is exercised. `example.spec.ts` still owns the real-backend smoke path.
- **Added** backend unit test suite — `apps/backend/src/errors/AppError.spec.ts`, `apps/backend/src/middleware/notFound.spec.ts`, `apps/backend/src/middleware/validate.spec.ts`, `apps/backend/src/middleware/errorHandler.spec.ts`. 17 tests across the four pure-function surfaces the backend previously only exercised through DB-backed e2e. Also added `apps/backend/src/types.d.ts` with a side-effect import of `pino-http` so ts-jest picks up the `http.IncomingMessage` module augmentation when compiling specs in isolation (the app build already imports `pino-http` from `main.ts`; the spec tsconfig doesn't see that transitively).
- **Added** frontend unit tests for `apps/frontend/src/components/UsersList.tsx` (4 render-branch tests: empty/populated/singular-copy/total-vs-slice) and `apps/frontend/src/lib/api/users.ts` (5 tests: query-key shape, happy-path fetch, three Zod rejection branches). Replaces the placeholder `__tests__/index.spec.tsx` that only asserted `true === true`. `fetchUsers` is now exported so it can be tested directly; the public hook `useUsersQuery` is unchanged.
- **Added** liveness/readiness split on the backend health endpoint (`apps/backend/src/routes/health.ts`). `/api/health/live` returns `200` with `{ status: "ok" }` without touching the DB — safe for orchestrator liveness probes that restart on failure. `/api/health/ready` keeps the existing DB-ping behaviour (`200` on success, `503` on failure) — this is what belongs on readiness probes that stop routing traffic. `/api/health` is retained as an alias of `/ready` for back-compat. Playwright's `webServer` wait and the README deployment checklist updated to point at `/ready` explicitly; `apps/backend-e2e/src/backend/backend.spec.ts` covers all three endpoints.
- **Added** `CODE_OF_CONDUCT.md` — adopts the Contributor Covenant 2.1 (linked to the canonical hosted text so it can't drift). Reporting contact is the maintainer's email (`victorvinci@protonmail.com`); security issues still route through `SECURITY.md`. Linked from `README.md` TOC and `CONTRIBUTING.md`.

### Security

- **Fixed** I1 from `docs/SECURITY-AUDIT-v1.0.0.md` — prominent "SECURITY INVARIANT" banner at the top of `.github/workflows/force-draft.yml` spelling out why `pull_request_target` is safe today (no PR-code checkout, no PR-authored string interpolation) and what additions would break that invariant. No behavioral change; the banner exists so a future contributor (or AI agent) doesn't silently turn this workflow into an RCE vector against the base repo.
- **Fixed** I2 from `docs/SECURITY-AUDIT-v1.0.0.md` — `README.md` Production Deployment step 5 now explains what `trust proxy: 1` means, why it's only correct for exactly one proxy hop, and what to change for zero-proxy or multi-proxy topologies. Wrong values let a client spoof `X-Forwarded-For` and bypass `express-rate-limit`; the doc previously did not warn operators about this.
- **Fixed** I3 from `docs/SECURITY-AUDIT-v1.0.0.md` — `attribution-guard` in `.github/workflows/ci.yml` now pipes each line newly added to `.ai-attribution.jsonl` through `jq -c . >/dev/null` and fails the PR on a parse error. Previously the guard only counted net-added lines, so an AI agent could have satisfied it by appending `{}` or a truncated object — the audit log's integrity now matches what `CLAUDE.md` promises.
- **Fixed** L4 from `docs/SECURITY-AUDIT-v1.0.0.md` — client-supplied `x-request-id` in `apps/backend/src/main.ts` is now validated against `/^[a-zA-Z0-9-]{1,64}$/` before being echoed into logs and the response header; anything else gets a fresh UUID. Closes the log-injection (newlines), storage-bloat (long strings), and header-smuggling vectors a malicious client previously had.
- **Fixed** L5 from `docs/SECURITY-AUDIT-v1.0.0.md` — pinned `renovate` version in `.github/workflows/ci.yml` `renovate-config` job (`npx --yes --package renovate@43.136.0 --`). Previously unpinned, which would have executed whatever was on npm latest on the day of a hypothetical supply-chain compromise. `renovate.json`'s customManager was widened to match this file too so Renovate bumps the pin itself.
- **Fixed** L3 from `docs/SECURITY-AUDIT-v1.0.0.md` — `apps/backend/src/middleware/validate.ts` no longer mutates `req.body` / `req.params` via `Object.assign`. Parsed data is now exposed on `res.locals.validatedBody` / `validatedQuery` / `validatedParams` (mirroring the query branch); handlers read from `res.locals`, not `req`. Removes a latent prototype-pollution surface and ensures caller-extra fields Zod stripped can never leak back into handler logic.
- **Fixed** L1 from `docs/SECURITY-AUDIT-v1.0.0.md` — `scripts/extract-changelog-section.sh` now matches the version header with `index()` instead of an interpolated awk regex, so a tag containing `.`, `*`, `[`, etc. can no longer match unintended sections.
- **Fixed** L2 from `docs/SECURITY-AUDIT-v1.0.0.md` — `scripts/dev-setup-native.sh` now rejects `DB_PASSWORD` values containing a single quote, double quote, or backslash (previously only the single-quote case). Backslash is a MySQL escape char inside `'...'` under default `sql_mode`, so `\'` would previously break the `IDENTIFIED BY '${DB_PASSWORD}'` literal and set a wrong password.
- **Fixed** `docker-compose.yml` M1 + M2 from `docs/SECURITY-AUDIT-v1.0.0.md`. Split the MySQL root password off the app password (`MYSQL_ROOT_PASSWORD` now reads `${DB_ROOT_PASSWORD}` instead of reusing `${DB_PASSWORD}`) and bound the published MySQL port to `127.0.0.1` instead of `0.0.0.0`, so the dev DB is no longer reachable from any network the host is attached to and a leaked app credential no longer grants MySQL root on the dev container. The healthcheck was switched from `-u root` to `-u ${DB_USER}` so it stays green without needing the root password at all (also lets operators later set `MYSQL_RANDOM_ROOT_PASSWORD=yes` if they want). `.env.example` gains a `DB_ROOT_PASSWORD` entry with a comment noting it must differ from `DB_PASSWORD`.

### Fixed

- **Fixed** `apps/frontend-e2e/src/users.failure.spec.ts` — two latent bugs that surfaced the first time the suite ran in CI on PR #43. (1) The empty-state test filtered by `getByRole('status', { name: /No users yet/ })`, but ARIA's `status` role has `Name From: author` only, so `<p role="status">No users yet.</p>` has an empty accessible name and matched nothing across chromium/firefox/webkit; switched to `.filter({ hasText: … })`. (2) The Retry test's counter-based stub returned `calls === 1 ? error : seeded`, which didn't account for React Query's default `retry: 1` auto-retry — the second call silently served the seeded payload, so the error UI never rendered and the manual Retry was never reached. Stub now fails for `calls <= 2` (initial + auto-retry) and the `calls` assertion is tightened to `>= 3`.
- **Fixed** `renovate.json` `customManagers[0]` — migrated deprecated `fileMatch` to `managerFilePatterns` (with regex delimiters `/.../`). Renovate 43.x treats the old key as "Config migration necessary", which fails the `renovate-config` CI job under `--strict`. No behavioral change — the pattern still matches `.github/workflows/ci.yml` and `ci-scheduled.yml`.

### Changed

- **Changed** `CLAUDE.md` "Commit attribution rule" — added a note that the `Co-Authored-By:` trailer must be separated from the commit body by a blank line. commitlint's `footer-leading-blank` warns without it (observed on the previous commit in this branch); the warning is non-blocking but `git interpret-trailers` won't recognise the line as a footer, so trailer-aware tooling can skip it. `attribution-guard` is unaffected — it greps the whole message.
- **Changed** `CONTRIBUTING.md` "Before you open a PR" — added a bullet codifying the unit-test expectation for pure backend helpers (middleware, error classes, services without DB), pointing at `apps/backend/src/middleware/validate.spec.ts` as the reference pattern. Integration tests were already required via the adjacent bullet; the new rule makes the two layers explicit so future PRs don't skip one.
- **Changed** `docs/SEMVER.md` internal-surfaces section — the `/health` example now reads `/health/{live,ready}` to match the split landed in this Unreleased cycle. The policy is unchanged (backend URL shapes remain internal); the prose just no longer lies about the current endpoint layout.
- **Changed** `.ai-attribution.jsonl` schema — removed the `commit` field. Squash-merge into develop and rebase-merge into main both rewrite SHAs, so every entry written pre-merge ended up pointing at an unreachable SHA. The authoritative audit signal is now the `Co-Authored-By: claude-<model-id>` trailer (durable across both merge strategies). The JSONL still carries `date`, `model`, `scope`, `description`, `files`. Historical entries with the old `commit` field are preserved as-is per the append-only rule — corrections go in new entries, not edits. `CLAUDE.md` schema section and two-commit flow rewritten; `docs/RELEASE.md` attribution step simplified (no more `git rev-parse --short HEAD` capture).
- **Changed** `.github/workflows/ci.yml` — `attribution-guard` job now does trailer-based integrity checking. For every commit in the PR with an AI-assistant `Co-Authored-By` trailer (claude, Claude Code, GPT, Gemini, Copilot, Cursor, Devin, Codex — extensible), the PR must contain at least one net-new line in `.ai-attribution.jsonl`. Missing entries now fail the build instead of warning. Automation bots (dependabot, renovate, github-actions) are explicitly excluded. The guard also now runs on every non-draft PR (previously gated to `apps/`/`libs/` changes), since AI commits can touch docs, workflows, or configs and still require attribution.
- **Changed** `README.md` CI/CD section — rewrote the `attribution-guard` and `detect` bullets to reflect the new trailer-based enforcement (previous text said "logged as a warning but does not block the PR", which is no longer true). Linked the `VITE_API_URL` placeholder note to the `build-frontend` step in `.github/workflows/pages.yml` where the value is actually set. Added pointer to `docs/SEMVER.md` from the Contributing section so fork authors land on the public-surface contract before proposing a breaking change.
- **Changed** `SECURITY.md` — removed the "pre-1.0" language from the Supported versions section. The policy is the same (latest tagged release on `main` gets fixes); the pre-1.0 qualifier was set to expire on the v1.0.0 cut and is cleaner stated as fork-first template policy.
- **Changed** `.github/pull_request_template.md` — added CHANGELOG-entry and `.ai-attribution.jsonl`-append checkboxes to the Test plan so contributors see the two `attribution-guard` rules before CI red-Xs them.
- **Changed** `renovate.json` — added a `description` on the `@types/node` package rule explaining the `<25` cap (matches `engines.node >= 24`), so the next maintainer who bumps the Node engine major knows to raise the cap in the same PR.
- **Changed** `.agents/skills/` — replaced the content-identical copy of `.github/skills/` with a symlink (`.agents/skills → ../.github/skills`). The two directories were byte-identical duplicates — `.github/skills/` is the one touched by git history, and keeping them in sync by hand would silently drift the next time one was edited.
- **Changed** `.github/workflows/ci-scheduled.yml` Lighthouse step — wrapped the `lhci autorun` invocation with the same tracking-issue pattern used by bundle-size and license-scan. On assertion failure the job files or updates a `lighthouse` tracking issue (extracting failing audits from `.lighthouseci/assertion-results.json` when available) and re-raises the failure via a separate step so the run stays red; closes the stale issue on a clean pass. Previously a loud red cron run on a solo-maintainer repo was functionally silent — the tracking issue makes the regression impossible to miss.
- **Changed** `docs/RELEASE.md` step 8 — the manual `awk | gh release create` block is replaced by the new `.github/workflows/release.yml` automation. The doc now describes what the workflow does (CHANGELOG extraction + SBOM + release publish) and keeps the `scripts/extract-changelog-section.sh` invocation as a manual fallback recipe for the rare re-publish case.

### Fixed

- **Fixed** `docs/RELEASE.md` "Hotfix flow" step 5 — replaced the underspecified "back-merge `main` → `develop`" instruction with an explicit cherry-pick recipe. A plain `git merge` is rejected by develop's `required_linear_history` ruleset, and the release-PR fast-forward trick (`push origin/main:develop`) would discard develop's un-merged work when develop has diverged during the hotfix. The new step describes a `hotfix-sync/<slug>` branch that cherry-picks the hotfix commit(s) from `main` into a squash-merge PR against develop. `.github/PULL_REQUEST_TEMPLATE/hotfix.md` cross-reference and post-merge checkbox also updated.
- **Fixed** `.github/workflows/ci-scheduled.yml` license-scan step — removed the `if: steps.lc.outputs.exit_code == '0'` gate that silently skipped the entire "Open or update tracking issue" action when `license-checker-rseidelsohn` errored (registry flake, npx install failure). Now always runs and splits into two paths: scanner-errored files a separate `license-scan-error` tracking issue (distinct label) and fails the step loudly; scanner-succeeded runs the original offender-finding flow and also closes any stale `license-scan-error` issue from a prior failed week. The normal `license-scan` issue is never touched on scanner failure, because a broken scan ≠ "no offenders".
- **Fixed** `.github/workflows/ci.yml` — removed the dead `environment: main` attribute from the `ci-pass` job. A prior version gated it as a pre-merge deploy environment; that rule was removed from the `main` ruleset (documented in `docs/RELEASE.md` → "Things this flow deliberately avoids") but the workflow wiring wasn't cleaned up. It now rendered as a cosmetic "Deployed to: main" tag in the run UI without gating anything.

### Changed

- **Changed** `.github/workflows/ci.yml` `attribution-guard` job — replaced `fetch-depth: 0` (full history clone) with `fetch-depth: 50` plus a deepening loop that fetches the PR base SHA and deepens in 50-commit increments until it's an ancestor of HEAD (capped at 10 deepens, then falls back to `--unshallow`). Saves seconds on this repo and minutes on forks with long histories, while still covering the `$BASE_SHA..HEAD` range that `attribution-guard` needs.
- **Changed** `.github/workflows/ci-scheduled.yml` top-level `concurrency:` — flipped `cancel-in-progress` from `false` to `true`. Two back-to-back `workflow_dispatch` runs (or a dispatch overlapping the cron) would otherwise race on the same tracking issues; every job here is idempotent so cancellation is safe.

## [0.2.0] - 2026-04-19

### Added

- **Added** `docs/RELEASE.md` — documents the end-to-end release workflow (branch from develop, two-commit AI-attribution flow, squash into develop, rebase into main, tag + GitHub release, post-release sync back to develop). Codifies the rules enforced by the `develop`, `main`, and `release-tags` rulesets so future releases follow a single pattern.

### Changed

- **Changed** `docs/RELEASE.md` step 9 (post-release sync) — replaced the `git merge --no-ff origin/main` approach (which creates a merge commit and violates develop's `required_linear_history` rule) with `git push origin origin/main:develop --force-with-lease`, a one-shot fast-forward of develop to main's rebased tip. Requires admin bypass of `non_fast_forward` on develop — documented as a deliberate once-per-release action.
- **Changed** `develop` and `main` rulesets — removed `Convert PR to Draft` from `required_status_checks`. The workflow is `opened`-only and never reruns on `synchronize`, so treating it as a required check left new pushes blocked by an unsatisfiable "Expected" status. - **JWT stub from frontend `api.ts`** — `localStorage` token read, `Authorization` header injection, and the 401 interceptor are gone. The backend never had matching auth, so this was misleading dead code. Re-add real auth (JWT, sessions, OAuth, Auth.js, …) when you need it.
- **`JWT_SECRET` / `SESSION_SECRET`** from `.env.example`.
- Unused `Link` import in `apps/frontend/src/routes/__root.tsx` (surfaced by the new strict-mode `noUnusedLocals`).

### Security

- **Stack-trace leakage fixed.** Production error responses now return a generic message; details stay in server logs.
- **CORS hardened.** No wildcard, no localhost fallback — `FRONTEND_URL` is required and validated by Zod.
- **Rate limiting added.** 100 req/min/IP globally; tune per-route as your traffic grows.
- **Body size capped** at 100 KB to prevent memory-exhaustion attacks. Raise per-route only if needed.
- **Helmet CSP made explicit** instead of relying on permissive defaults.
- **`x-powered-by: Express` header removed** so the framework isn't advertised to attackers.
- **Env validation at boot** — the app fails fast on missing or malformed config rather than starting in a broken state.
- **`mysql2` configured with `namedPlaceholders: true`** — combined with the existing parameterized queries, this keeps the boilerplate's SQL safe by default. Document new code must follow suit.
- **Outstanding `npm audit` finding documented.** Direct `axios` is on the latest version (`1.15.0`); the 7 critical CVEs all live under transitive dev dependencies (`@module-federation/*` via `@nx/react`, `jsdom` via `@tootallnate/once`). They are **not** shipped to production bundles. `npm audit fix --force` would jump `jsdom` to v29 and break Vitest, so the upgrade is deferred until upstream releases. Re-check on every release.
- **No authentication is included.** The README and CLAUDE.md now state this explicitly. Forks must add their own auth layer before exposing protected data.
